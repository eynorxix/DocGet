### `tsconfig.json` (Configuración de TypeScript)

#### Propósito
Este archivo configura el compilador de TypeScript para el proyecto. Define cómo se compilan los archivos TypeScript, qué reglas se aplican y cómo se resuelven los módulos, asegurando la compatibilidad con el entorno de Expo.

#### Estructura
Es un objeto JSON que extiende la configuración base de Expo y define opciones específicas del compilador, rutas de módulos y archivos a incluir/excluir.

#### Detalles Técnicos
*   **`extends`**: `"expo/tsconfig.base"`. Hereda la configuración base recomendada por Expo para proyectos TypeScript, lo que simplifica la configuración inicial y asegura la compatibilidad.
*   **`compilerOptions.strict`**: `true`. Habilita todas las comprobaciones de tipo estrictas de TypeScript, promoviendo un código más robusto y menos propenso a errores.
*   **`compilerOptions.paths`**: Define alias para rutas de importación, mejorando la organización del código y facilitando las importaciones:
    *   `@/*`: Mapea a `./src/*`.
    *   `@/assets/*`: Mapea a `./assets/*`.
    *   `@modulos/*`: Mapea a `./modulos/*`.
*   **`include`**: Lista de patrones de archivos que el compilador debe incluir:
    *   `**/*.ts` y `**/*.tsx`: Todos los archivos TypeScript y TSX en el proyecto.
    *   `.expo/types/**/*.ts`: Tipos específicos de Expo.
    *   `expo-env.d.ts`: Archivo de declaración de entorno de Expo.
*   **`exclude`**: `modulos/testing`. Excluye la carpeta de `testing` dentro de `modulos` del proceso de compilación principal, probablemente porque contiene código de prueba que no es parte de la aplicación final o tiene configuraciones de compilación separadas.

#### Notas Importantes
*   La extensión de `expo/tsconfig.base` es crucial para la integración con el ecosistema de Expo.
*   El uso de `strict: true` es una buena práctica para mantener la calidad del código TypeScript.
*   Los `paths` facilitan la modularidad y evitan rutas relativas largas y propensas a errores.
*   La exclusión del directorio `modulos/testing` sugiere que las pruebas tienen su propio manejo o no deben ser compiladas con la aplicación principal.

### `pnpm-workspace.yaml` (Configuración del Espacio de Trabajo pnpm)

#### Propósito
Este archivo es parte de la configuración del gestor de paquetes pnpm para un espacio de trabajo (monorepo). Aunque en este caso no define múltiples proyectos dentro de un espacio de trabajo (la sección `packages` está ausente), incluye configuraciones de construcción específicas para ciertas dependencias.

#### Estructura
Es un archivo YAML que contiene claves de configuración para pnpm.

#### Detalles Técnicos
*   **`allowBuilds`**: Una lista de paquetes para los cuales pnpm debe permitir la fase de construcción (normalmente para paquetes nativos o con scripts de post-instalación). Esto es útil para dependencias que requieren compilación en el entorno local.
    *   `'@firebase/util'`: `true`
    *   `msgpackr-extract`: `true`
    *   `protobufjs`: `true`
    *   `unrs-resolver`: `set this to true or false` (indica una configuración pendiente).
*   **`minimumReleaseAge`**: `0`. Especifica la edad mínima (en segundos) que un paquete debe tener en el registro para ser considerado para una nueva instalación, aunque el valor `0` desactiva efectivamente esta restricción.

#### Notas Importantes
*   La sección `allowBuilds` es importante para garantizar que las dependencias con código nativo o que necesitan compilación se instalen correctamente.
*   El `AGENTS.md` especifica el uso exclusivo de `pnpm`, haciendo de este archivo una parte fundamental de la gestión de dependencias del proyecto.
*   La entrada para `unrs-resolver` indica una decisión de configuración pendiente que podría afectar la construcción de esa dependencia.

### `pnpm-lock.yaml` (Archivo de Bloqueo de pnpm)

#### Propósito
Este archivo de bloqueo (`lockfile`) garantiza que las dependencias del proyecto se instalen de manera consistente en cualquier entorno. Registra las versiones exactas y la estructura del árbol de dependencias, incluyendo las dependencias transitivas.

#### Estructura
Es un archivo YAML que especifica la versión del lockfile, configuraciones globales y, lo más importante, las dependencias exactas y sus resoluciones para cada paquete en el espacio de trabajo (en este caso, el proyecto raíz).

#### Detalles Técnicos
*   **`lockfileVersion`**: `'9.0'`. Indica la versión del formato del archivo de bloqueo de pnpm, lo que es relevante para la compatibilidad con diferentes versiones de pnpm.
*   **`settings`**:
    *   `autoInstallPeers`: `true`. pnpm instalará automáticamente las dependencias peer que falten.
    *   `excludeLinksFromLockfile`: `false`. Los enlaces simbólicos se incluirán en el lockfile.
*   **`importers`**:
    *   `.` (proyecto raíz): Contiene la lista de dependencias directas del proyecto.
    *   **`dependencies`**: Cada entrada especifica el `specifier` (la versión declarada en `package.json`) y la `version` exacta instalada (incluyendo las resoluciones y sus dependencias peer entre paréntesis).
        *   Ejemplo: `expo: 56.0.8(@babel/core@7.29.7)(...)` muestra la versión de Expo y una lista exhaustiva de sus dependencias peer resueltas y sus versiones específicas. Esto es crucial para la reproducibilidad.

#### Notas Importantes
*   El `pnpm-lock.yaml` es esencial para la integridad y reproducibilidad de las builds, asegurando que todos los desarrolladores y entornos de CI/CD usen las mismas versiones de las dependencias.
*   La detallada información de las versiones de las dependencias (ej. `expo` con sus dependencias peer anidadas) es una característica clave de pnpm que garantiza un árbol de dependencias determinista y optimizado.
*   Este archivo no debe ser editado manualmente; pnpm lo genera y actualiza automáticamente.

### `jest.config.js` (Configuración de Jest)

#### Propósito
Este archivo configura el framework de pruebas Jest para el proyecto. Define cómo se descubren, ejecutan y procesan los archivos de prueba, así como cómo se manejan los módulos y las transformaciones de código.

#### Estructura
Es un archivo JavaScript que exporta un objeto de configuración de Jest.

#### Detalles Técnicos
*   **`testMatch`**: `['**/modulos/testing/**/*.test.ts']`. Especifica que Jest solo debe buscar archivos de prueba con la extensión `.test.ts` dentro de la carpeta `modulos/testing`. Esto centraliza las pruebas en un lugar específico.
*   **`moduleFileExtensions`**: `['ts', 'tsx', 'js', 'jsx']`. Define las extensiones de archivo que Jest debe reconocer como módulos.
*   **`moduleNameMapper`**: Objeto para mapear rutas de importación a archivos simulados o rutas reales. Esto es crucial para el testing, permitiendo aislar módulos o mockear dependencias externas:
    *   `'^@/lib/firestore$'`: Mapea a un mock de `firestore-lib.ts`.
    *   `'^@/(.*)$'`: Mapea a `<rootDir>/src/$1`.
    *   `'^@assets/(.*)$'`: Mapea a `<rootDir>/assets/$1`.
    *   `'^@modulos/(.*)$'`: Mapea a `<rootDir>/modulos/$1`. Estos alias coinciden con los definidos en `tsconfig.json`.
    *   `'^firebase/firestore$'`: Mapea a un mock de `firestore.ts`.
    *   `'^react-native$'`: Mapea a un mock de `react-native.ts`.
*   **`transform`**: Define cómo Jest debe procesar archivos antes de ejecutar las pruebas.
    *   `'^.+\\.tsx?$': ['ts-jest', { diagnostics: false }]`. Utiliza `ts-jest` para transformar archivos TypeScript y TSX a JavaScript, con `diagnostics: false` para evitar que `ts-jest` imprima advertencias de diagnóstico de TypeScript durante las pruebas.
*   **`transformIgnorePatterns`**: `[]`. Una lista de patrones de regex para ignorar la transformación de módulos. Una matriz vacía significa que no se ignoran patrones específicos, lo cual es importante para que `ts-jest` pueda transformar módulos de terceros si es necesario (común en proyectos React Native/Expo).
*   **`testEnvironment`**: `'node'`. Indica que las pruebas deben ejecutarse en un entorno Node.js, lo cual es adecuado para pruebas que no requieren un navegador o un entorno DOM real (aunque a menudo se usa `jsdom` para React). El uso de `node` aquí, junto con mocks para `react-native`, sugiere que las pruebas son más unitarias y lógicas que de renderizado de UI.

#### Notas Importantes
*   La configuración de `moduleNameMapper` es vital para el aislamiento de pruebas, permitiendo mockear servicios como Firebase o React Native, lo que asegura que las pruebas se centren en la lógica del código y no en dependencias externas.
*   El uso de `ts-jest` permite escribir pruebas directamente en TypeScript, manteniendo la coherencia del lenguaje en todo el proyecto.
*   El `testEnvironment: 'node'` y los mocks de `react-native` sugieren un enfoque hacia pruebas unitarias y de integración que no dependen de un entorno de renderizado completo de React Native, lo que puede acelerar la ejecución de pruebas.

### `babel.config.js` (Configuración de Babel)

#### Propósito
Este archivo configura Babel, un transpilador de JavaScript, para transformar el código fuente (por ejemplo, JSX, características modernas de JavaScript, TypeScript) a una versión compatible con los entornos de ejecución (dispositivos móviles, navegadores web) especificados por Expo.

#### Estructura
Es un archivo JavaScript que exporta una función que devuelve un objeto de configuración de Babel.

#### Detalles Técnicos
*   `module.exports = function (api)`: La configuración es una función que recibe un objeto `api`, permitiendo el acceso a utilidades de Babel.
*   `api.cache(true)`: Habilita el caché para la configuración de Babel, lo que acelera las compilaciones sucesivas al evitar la reevaluación innecesaria de la configuración.
*   `presets: ['babel-preset-expo']`: Utiliza el preset oficial de Babel para Expo. Este preset incluye las transformaciones necesarias para el desarrollo de aplicaciones con React Native y Expo, cubriendo JSX, ES6+, Flow/TypeScript (si se configura), y más.

#### Notas Importantes
*   El `babel-preset-expo` es fundamental para que el código moderno de React Native/Expo, incluyendo JSX y características de JavaScript avanzadas, sea compatible con los entornos de ejecución.
*   El caché mejora significativamente los tiempos de construcción, especialmente en proyectos grandes.
*   Este archivo es una parte estándar de un proyecto Expo/React Native y rara vez requiere modificaciones profundas a menos que se necesiten transformaciones muy específicas no cubiertas por el preset de Expo.

### Resumen General

Este conjunto de archivos define la infraestructura, el comportamiento y las directrices de desarrollo para un proyecto de aplicación móvil/web universal construido con Expo y React Native, utilizando TypeScript y pnpm como gestor de paquetes.

*   **`package.json`** es el corazón que define el proyecto (`proyectoii`, versión `1.0.0`), sus scripts operativos (`start`, `android`, `ios`, `web`, `test`) y todas sus dependencias. Se destaca el uso de **Expo SDK v56**, `react` v19, `react-native` v0.85 y `firebase`.

*   **`pnpm-workspace.yaml`** y **`pnpm-lock.yaml`** son fundamentales para la gestión de dependencias con `pnpm` (el gestor de paquetes exclusivo según `AGENTS.md`). `pnpm-workspace.yaml` define configuraciones de construcción para paquetes específicos, mientras que `pnpm-lock.yaml` asegura la reproducibilidad de las instalaciones de dependencias, detallando versiones exactas y sus árboles de dependencias.

*   **`tsconfig.json`** y **`expo-env.d.ts`** trabajan juntos para configurar el entorno de desarrollo TypeScript. `tsconfig.json` extiende la configuración base de Expo, habilita el modo estricto, y define alias de rutas (`@/`, `@assets/`, `@modulos/`) para una mejor organización del código. `expo-env.d.ts` importa los tipos de entorno específicos de Expo, asegurando la compatibilidad de tipos en todo el proyecto.

*   **`babel.config.js`** complementa la configuración de compilación transpilando el código moderno de JavaScript/TypeScript y JSX a una versión compatible con los entornos de ejecución, utilizando el `babel-preset-expo`.

*   **`app.json`** es el manifiesto de la aplicación Expo. Configura metadatos como el nombre (`ProyectoII`), la versión, los iconos, los esquemas de URL, y las configuraciones específicas para iOS, Android y Web. Es crucial la habilitación de **Expo Router** con `typedRoutes` y la `newArchEnabled`, lo que indica un proyecto moderno con navegación basada en archivos y potencial mejora de rendimiento.

*   **`jest.config.js`** establece el entorno de pruebas. Define dónde buscar las pruebas (`**/modulos/testing/**/*.test.ts`), cómo transformar el código (`ts-jest`) y, lo más importante, cómo mapear y mockear módulos (`@/lib/firestore`, `firebase/firestore`, `react-native`) para un testing aislado y eficiente.

*   Finalmente, **`AGENTS.md`** (referenciado por `CLAUDE.md`) actúa como el documento rector del proyecto. Establece directrices clave como el uso exclusivo de `pnpm`, comandos de ejecución, la versión de Expo SDK, un tema de diseño `dark mode` con colores centralizados en `src/constants/theme.ts` (prohibiendo colores hardcodeados) y directrices exhaustivas para la documentación del proyecto.

En conjunto, estos archivos forman un ecosistema bien definido para el desarrollo de una aplicación Expo, enfocada en la robustez (TypeScript estricto, pruebas), la eficiencia (pnpm, caché de Babel), la modernidad (Expo Router, New Architecture, React Compiler experimental) y la mantenibilidad (directrices de estilo y documentación).
